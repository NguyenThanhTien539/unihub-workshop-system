# ADR-005: Use Local Mobile Database for Offline Check-in

## Status

Accepted

## Context

Check-in staff use the mobile app to scan QR codes at room entrances. Some areas in the university may have unstable or unavailable network connectivity. However, staff still need to continue recording attendance during the event.

The system must avoid losing check-in data when:

- the network is temporarily unavailable,
- the app is closed or restarted,
- multiple scans happen while offline,
- the same QR code is scanned more than once,
- multiple devices later synchronize overlapping check-in events.

Online-only check-in would not satisfy the requirement because staff would be blocked whenever the network is unavailable.

## Decision

Use a local mobile database, SQLite, inside the React Native check-in app.

SQLite stores:

- cached workshop/session metadata,
- cached validation metadata where appropriate,
- offline check-in events,
- sync queue state,
- sync status for each event.

Each offline check-in event has a unique `sync_event_id` generated by the mobile app. When connectivity returns, the app synchronizes queued events with the Backend API. The backend processes sync events idempotently using `sync_event_id` and registration/session constraints.

Offline check-ins are treated as provisional on the device. The backend remains the final source of truth for whether a check-in is accepted, duplicated, or rejected.

## Rationale

SQLite provides durable local persistence on mobile devices. Unlike in-memory storage, it can survive app restarts, device sleep, and temporary network loss.

This is suitable for the check-in scenario because staff need to scan quickly at the room entrance without waiting for every request to reach the server. A local sync queue allows the app to store scan events immediately and submit them later.

Using `sync_event_id` makes synchronization idempotent. If the same event is sent multiple times due to retry, the backend can recognize it and avoid creating duplicate records. Backend constraints such as unique check-in per registration provide a second layer of protection when two devices scan the same student.

## Consequences

Positive consequences:

- Check-in can continue during network outages.
- Unsynced check-in events survive app restarts.
- The mobile app can show immediate provisional feedback to staff.
- The backend can safely deduplicate repeated sync attempts.
- The design supports eventual consistency between mobile devices and the server.

Trade-offs and risks:

- The mobile app needs sync queue management.
- The system must handle conflicts when multiple devices scan the same registration.
- Offline validation may be incomplete or stale compared to backend validation.
- Staff must periodically refresh cached session data before or during the event.
- Local data must be protected because it may contain student attendance information.
- The UI must clearly distinguish provisional offline check-ins from backend-confirmed check-ins.

## Failure and Conflict Handling

- If the same QR code is scanned twice on the same device while offline, the local database should mark it as duplicate or prevent creating a second pending event.
- If the same registration is checked in by two devices, the backend accepts the first valid check-in and returns `duplicate` for later events.
- If a QR code is invalid, revoked, expired, or does not belong to the selected session, the backend returns `rejected` during synchronization.
- If synchronization fails, the event remains in SQLite with status `PENDING_SYNC` or `SYNC_FAILED` and can be retried later.
- If the app is restarted, unsynced events remain available in SQLite.
- If cached session data is stale, backend validation overrides the local result during synchronization.

## Alternatives Considered

### In-memory offline storage only

In-memory storage was rejected because app restarts, crashes, or device sleep could lose unsynced check-in events.

### Local file or JSON storage

Local JSON/file storage was rejected because it is weaker for queryability, sync status tracking, duplicate detection, and transactional updates compared to SQLite.

### Server-only check-in

Server-only check-in was rejected because it fails during network outages. This does not satisfy the project requirement for offline check-in.

### Browser localStorage-based PWA storage

A PWA storage approach was considered, but React Native with SQLite provides a more reliable mobile experience for camera scanning, durable storage, and offline queue management.
